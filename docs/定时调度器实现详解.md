# AgentScope 定时调度器实现详解

## 概述

AgentScope 调度器扩展位于 `agentscope-extensions/agentscope-extensions-scheduler`，为 Agent 提供定时调度执行能力。当前提供 **Quartz**（单机/集群）和 **XXL-Job**（分布式）两种实现，均基于统一的 `AgentScheduler` 接口。

## 模块结构

```
agentscope-extensions-scheduler/
├── agentscope-extensions-scheduler-common/     # 核心接口与配置
│   ├── AgentScheduler                          # 调度器统一接口
│   ├── ScheduleAgentTask                       # 任务接口
│   ├── BaseScheduleAgentTask                   # 任务基类（每次执行创建新 Agent）
│   └── config/                                 # 配置类
│       ├── AgentConfig / RuntimeAgentConfig    # Agent 配置（元数据 vs 运行时）
│       ├── ModelConfig / DashScopeModelConfig  # 模型配置
│       ├── ScheduleConfig / ScheduleMode       # 调度策略配置
│       └── ScheduleMode                        # NONE / CRON / FIXED_RATE / FIXED_DELAY
├── agentscope-extensions-scheduler-quartz/     # Quartz 实现
│   ├── QuartzAgentScheduler                    # 主调度器（支持全部调度模式）
│   ├── AgentQuartzJob                          # Quartz Job 实现
│   ├── QuartzScheduleAgentTask                 # 支持 pause/resume
│   └── QuartzAgentSchedulerRegistry            # 静态注册表
└── agentscope-extensions-scheduler-xxl-job/    # XXL-Job 实现
    └── XxlJobAgentScheduler                    # 分布式调度器
```

## 一、Quartz 实现

### 实现方式

`QuartzAgentScheduler` 基于 Quartz Scheduler，使用 Builder 模式创建，**支持全部四种调度模式**。

| 调度模式 | 说明 | 底层实现 |
|----------|------|----------|
| `CRON` | Cron 表达式定时 | `CronTrigger`，支持时区 |
| `FIXED_RATE` | 固定频率执行 | `SimpleTrigger`，`repeatForever()` |
| `FIXED_DELAY` | 固定延迟执行（上一次完成后延迟指定时间） | 一次性触发 + 完成后手动重新调度 |
| `NONE` | 仅注册不调度 | 注册 Job 但不创建 Trigger |

### 核心类职责

- **`QuartzAgentScheduler`**：主调度器，实现 `AgentScheduler` 接口，提供 `schedule`/`cancel`/`pause`/`resume`/`interrupt`/`getStatus`/`shutdown` 等完整生命周期管理。
- **`AgentQuartzJob`**：实现 Quartz 的 `InterruptableJob` 接口，通过 `QuartzAgentSchedulerRegistry` 查找调度器和任务，调用 `task.run().block()` 执行 Agent，并处理 `FIXED_DELAY` 的重新调度逻辑。
- **`QuartzScheduleAgentTask`**：扩展 `BaseScheduleAgentTask`，封装 Quartz `JobKey`，提供 `pause()`/`resume()` 能力。
- **`QuartzAgentSchedulerRegistry`**：基于 `ConcurrentHashMap` 的静态注册表，将调度器 ID 映射到 `QuartzAgentScheduler` 实例。

### 使用示例

```java
// 创建调度器
AgentScheduler scheduler = QuartzAgentScheduler.builder()
    .autoStart(true)
    .build();

// 配置并注册任务
ModelConfig modelConfig = DashScopeModelConfig.builder()
    .apiKey(apiKey).modelName("qwen-plus").build();

AgentConfig agentConfig = AgentConfig.builder()
    .name("MyLocalAgent")
    .modelConfig(modelConfig)
    .sysPrompt("You are a helpful assistant")
    .build();

// CRON 方式：每天早上 8 点执行
ScheduleConfig scheduleConfig = ScheduleConfig.builder()
    .scheduleMode(ScheduleMode.CRON)
    .cronExpression("0 0 8 * * ?")
    .zoneId("Asia/Shanghai")
    .build();

ScheduleAgentTask task = scheduler.schedule(agentConfig, scheduleConfig);

// 生命周期管理
scheduler.pause("MyLocalAgent");
scheduler.resume("MyLocalAgent");
scheduler.cancel("MyLocalAgent");
scheduler.shutdown();
```

### 特点总结

- **功能最完整**：支持所有调度模式和完整的生命周期管理（暂停/恢复/中断/状态查询）
- **可嵌入式**：无需外部依赖服务，适合单机或小规模集群场景
- **支持持久化**：可注入自定义 Quartz `Scheduler` 实例，配合数据库存储实现集群调度

## 二、XXL-Job 实现

### 实现方式

`XxlJobAgentScheduler` 基于 XXL-Job 分布式任务调度平台。构造函数接收一个已初始化的 `XxlJobExecutor` 实例，将 Agent 注册为 XXL-Job 的 `IJobHandler`。

| 特性 | 说明 |
|------|------|
| 调度模式 | **仅支持 `NONE`**（其他模式抛出 `UnsupportedOperationException`） |
| 取消任务 | **不支持**（抛出 `UnsupportedOperationException`） |
| 实际调度 | 由 XXL-Job Admin 控制台配置 Cron 表达式和执行策略 |
| 运行日志 | 自动添加 Hook 捕获 Agent 输出，通过 `XxlJobHelper.log()` 记录到 XXL-Job |

### 核心设计

1. **`schedule()`** 方法将 Agent 注册为 XXL-Job 的 JobHandler，Agent 名称即为 JobHandler 名称。
2. **执行时**动态创建 `ReActAgent` 实例，将 XXL-Job 传入的任务参数作为用户消息发送给 Agent，阻塞等待结果。
3. **日志集成**：自动添加 `PostReasoningEvent` Hook，将 Agent 的推理输出记录到 XXL-Job 日志系统。

### 使用示例

```java
// 初始化 XXL-Job Executor
XxlJobExecutor executor = new XxlJobExecutor();
executor.setAdminAddresses("http://localhost:8080/xxl-job-admin");
executor.setAppname("agentscope-demo");
executor.setPort(9999);
executor.start();

// 创建调度器
AgentScheduler scheduler = new XxlJobAgentScheduler(executor);

// 注册 Agent（调度策略在 XXL-Job 控制台配置）
RuntimeAgentConfig agentConfig = RuntimeAgentConfig.builder()
    .name("daily_report_agent")
    .modelConfig(DashScopeModelConfig.builder()
        .apiKey(apiKey).modelName("qwen-plus").build())
    .sysPrompt("你是一个日报生成助手")
    .toolkit(toolkit)  // 绑定工具集（如发送通知、生成报告等）
    .build();

ScheduleConfig scheduleConfig = ScheduleConfig.builder()
    .scheduleMode(ScheduleMode.NONE)  // XXL-Job 只支持 NONE
    .build();

scheduler.schedule(agentConfig, scheduleConfig);
// 之后在 XXL-Job Admin 控制台中配置 Cron 表达式
```

### 特点总结

- **分布式调度**：适合大规模、多节点部署场景
- **集中管理**：所有任务的调度策略在 XXL-Job Admin 控制台统一配置和管理
- **功能受限**：不支持通过代码动态设置调度模式/取消任务，这些操作需在管理控制台完成
- **依赖外部服务**：需要独立部署 XXL-Job Admin

## 三、两种实现对比

| 特性 | Quartz | XXL-Job |
|------|--------|---------|
| **部署方式** | 嵌入式，无需外部服务 | 需要独立部署 Admin 服务端 |
| **调度模式** | CRON / FIXED_RATE / FIXED_DELAY / NONE | 仅 NONE（调度由控制台配置） |
| **暂停/恢复** | 支持 | 不支持 |
| **取消任务** | 支持 | 不支持（需在控制台操作） |
| **中断执行** | 支持 | 不支持 |
| **状态查询** | 支持（返回 TriggerState） | 不支持 |
| **分布式** | 需配置数据库集群 | 原生支持，通过 Admin 协调 |
| **运行日志** | 自行管理 | 自动集成到 XXL-Job 日志系统 |
| **工具集绑定** | 通过 RuntimeAgentConfig | 通过 RuntimeAgentConfig |
| **适用场景** | 单机应用、嵌入式调度 | 微服务、分布式集群 |

## 四、智能体对话创建定时任务能力分析

### 当前状态：不支持

当前调度器扩展**不支持**通过智能体对话方式直接创建定时任务。原因如下：

1. **无 `@Tool` 注解**：整个调度器扩展模块中没有定义任何 `@Tool` 注解的方法，智能体无法在对话中调用调度 API。
2. **纯编程 API**：`AgentScheduler` 接口是面向开发者的编程 API，需在 Java 代码中显式调用 `schedule()`/`cancel()` 等方法。
3. **现有示例验证**：在 `boba-tea-shop` 示例项目中，定时任务是在 Spring `@Configuration` 类中编程式注册的，`ScheduleAgentTools` 中的 `@Tool` 方法是供**被调度的 Agent 执行时使用**的工具（如获取日报数据、发送报告），而非创建/管理调度任务本身的工具。

### 如何实现对话式创建定时任务

要实现通过智能体对话创建定时任务的能力，需要开发一个**调度管理工具类**，将 `AgentScheduler` 的能力暴露为 `@Tool` 方法，并注册到对话 Agent 的 Toolkit 中。示例设计如下：

```java
@Component
public class ScheduleManagementTools {

    private final AgentScheduler scheduler;

    public ScheduleManagementTools(AgentScheduler scheduler) {
        this.scheduler = scheduler;
    }

    @Tool(name = "create_scheduled_task", description = "创建一个定时任务，让 Agent 按指定策略周期性执行")
    public String createScheduledTask(
            @ToolParam("任务名称，唯一标识") String name,
            @ToolParam("Agent 的系统提示词，描述任务目标和执行流程") String sysPrompt,
            @ToolParam("调度模式：CRON / FIXED_RATE / FIXED_DELAY") String scheduleMode,
            @ToolParam("调度参数：CRON 模式填 cron 表达式，FIXED_RATE/FIXED_DELAY 模式填毫秒数") String scheduleParam
    ) {
        ScheduleConfig scheduleConfig = ScheduleConfig.builder()
            .scheduleMode(ScheduleMode.valueOf(scheduleMode))
            .build();

        // 根据 scheduleMode 设置具体参数
        switch (ScheduleMode.valueOf(scheduleMode)) {
            case CRON -> scheduleConfig = ScheduleConfig.builder()
                .scheduleMode(ScheduleMode.CRON)
                .cronExpression(scheduleParam)
                .build();
            case FIXED_RATE -> scheduleConfig = ScheduleConfig.builder()
                .scheduleMode(ScheduleMode.FIXED_RATE)
                .fixedRate(Long.parseLong(scheduleParam))
                .build();
            case FIXED_DELAY -> scheduleConfig = ScheduleConfig.builder()
                .scheduleMode(ScheduleMode.FIXED_DELAY)
                .fixedDelay(Long.parseLong(scheduleParam))
                .build();
        }

        AgentConfig agentConfig = AgentConfig.builder()
            .name(name)
            .modelConfig(/* 预配置的模型 */)
            .sysPrompt(sysPrompt)
            .build();

        ScheduleAgentTask task = scheduler.schedule(agentConfig, scheduleConfig);
        return "定时任务 '" + name + "' 创建成功";
    }

    @Tool(name = "cancel_scheduled_task", description = "取消一个已创建的定时任务")
    public String cancelScheduledTask(
            @ToolParam("要取消的任务名称") String name
    ) {
        boolean cancelled = scheduler.cancel(name);
        return cancelled ? "任务 '" + name + "' 已取消" : "取消失败，任务不存在";
    }

    @Tool(name = "list_scheduled_tasks", description = "列出所有已创建的定时任务")
    public String listScheduledTasks() {
        return scheduler.getAllScheduleAgentTasks().stream()
            .map(t -> t.getName() + " (ID: " + t.getId() + ")")
            .collect(Collectors.joining("\n"));
    }

    @Tool(name = "pause_scheduled_task", description = "暂停一个正在运行的定时任务")
    public String pauseScheduledTask(
            @ToolParam("要暂停的任务名称") String name
    ) {
        if (scheduler instanceof QuartzAgentScheduler quartz) {
            quartz.pause(name);
            return "任务 '" + name + "' 已暂停";
        }
        return "当前调度器不支持暂停操作";
    }

    @Tool(name = "resume_scheduled_task", description = "恢复一个已暂停的定时任务")
    public String resumeScheduledTask(
            @ToolParam("要恢复的任务名称") String name
    ) {
        if (scheduler instanceof QuartzAgentScheduler quartz) {
            quartz.resume(name);
            return "任务 '" + name + "' 已恢复";
        }
        return "当前调度器不支持恢复操作";
    }
}
```

将上述工具注册到对话 Agent 的 Toolkit 中后，用户即可通过自然语言与 Agent 对话来创建和管理定时任务，例如：

> 用户：每天早上 8 点帮我生成一份项目日报
> Agent：（调用 `create_scheduled_task` 工具）好的，已为您创建每天早上 8 点执行的日报生成任务。

> 用户：暂停日报任务
> Agent：（调用 `pause_scheduled_task` 工具）已暂停日报任务。

### XXL-Job 方案：通过 Admin API 实现完整的对话式动态创建

当前 `XxlJobAgentScheduler` 的 `cancel()` 抛出 `UnsupportedOperationException`，调度策略也需在控制台配置。但 **XXL-Job 本身完全支持通过 API 动态管理任务**，只需在 `@Tool` 工具类中封装 XXL-Job Admin 的 HTTP 接口即可补齐这个能力。

#### 方案一：调用 XXL-Job Admin 内部 HTTP 接口（推荐）

XXL-Job Admin 的管理后台本身就是通过 HTTP 接口操作任务的，这些接口可以直接被程序调用。

##### 认证机制说明：不需要手动登录

XXL-Job Admin 的 `/jobinfo/*` 接口受 `@PermissionLimit` 注解保护，需要 Cookie 认证。但**登录过程完全可以自动化**，整个流程如下：

```
应用启动
    │
    ▼
POST /login?userName=admin&password=123456   ← 代码自动发起，无需人工操作
    │
    ▼
响应 Set-Cookie: XXL_JOB_LOGIN_IDENTITY=xxxxx   ← 自动提取 Cookie
    │
    ▼
后续所有请求自动带上 Cookie: XXL_JOB_LOGIN_IDENTITY=xxxxx
    │
    ▼
（可选）Cookie 过期时自动重新登录
```

**关键点**：
- 登录只是一个 `POST /login` 请求，传入 `userName` 和 `password`，返回 `XXL_JOB_LOGIN_IDENTITY` Cookie
- **全程代码完成，不需要人工打开浏览器登录**
- Cookie 有过期时间（默认与 Session 超时一致），可通过拦截 302 重定向或检测响应状态自动重新登录
- 账号密码配置在 `application.yml` 中，随应用一起部署

**自动登录 + 自动续期的封装示例**：

```java
@Component
public class XxlJobAdminClient {

    private final String adminAddresses;
    private final String username;
    private final String password;
    private volatile String loginCookie;    // 缓存 Cookie，避免每次都登录

    @PostConstruct
    public void init() {
        // 应用启动时自动登录，获取 Cookie
        this.loginCookie = doLogin();
    }

    /**
     * 自动登录，返回 XXL_JOB_LOGIN_IDENTITY 的值
     */
    private String doLogin() {
        // POST {admin}/login?userName=xxx&password=xxx
        HttpResponse response = HttpRequest.post(adminAddresses + "/login")
            .form("userName", username)
            .form("password", password)
            .execute();

        // 从响应中提取 XXL_JOB_LOGIN_IDENTITY Cookie
        String cookie = response.getCookies().stream()
            .filter(c -> "XXL_JOB_LOGIN_IDENTITY".equals(c.getName()))
            .map(HttpCookie::getValue)
            .findFirst()
            .orElseThrow(() -> new RuntimeException("XXL-Job 登录失败"));

        log.info("XXL-Job Admin 登录成功");
        return cookie;
    }

    /**
     * 带自动续期的请求方法：如果 Cookie 过期（302 重定向到登录页），自动重新登录
     */
    private <T> T requestWithAutoRelogin(Supplier<T> requestFn) {
        try {
            T result = requestFn.get();
            // 如果被重定向到登录页，说明 Cookie 过期
            return result;
        } catch (Exception e) {
            // Cookie 过期，自动重新登录后重试
            log.warn("XXL-Job Cookie 过期，自动重新登录...");
            this.loginCookie = doLogin();
            return requestFn.get();  // 重试
        }
    }

    // ==================== 业务 API ====================

    public JSONObject addJob(Map<String, String> params) {
        return requestWithAutoRelogin(() ->
            doPost("/jobinfo/add", params));
    }

    public void startJob(int jobId) {
        requestWithAutoRelogin(() -> doGet("/jobinfo/start?id=" + jobId));
    }

    public void stopJob(int jobId) {
        requestWithAutoRelogin(() -> doGet("/jobinfo/stop?id=" + jobId));
    }

    public void removeJob(int jobId) {
        requestWithAutoRelogin(() -> doGet("/jobinfo/remove?id=" + jobId));
    }

    public JSONObject updateJob(Map<String, String> params) {
        return requestWithAutoRelogin(() ->
            doPost("/jobinfo/update", params));
    }

    public JSONObject pageList(Map<String, String> params) {
        return requestWithAutoRelogin(() ->
            doPost("/jobinfo/pageList", params));
    }
}
```

##### Admin HTTP 接口列表

| 接口 | 方法 | 说明 |
|------|------|------|
| `{admin}/login` | POST | 登录获取 Cookie（代码自动调用，无需人工） |
| `{admin}/jobinfo/add` | POST | 创建任务（含 Cron、执行器、JobHandler 等） |
| `{admin}/jobinfo/update` | POST | 更新任务（修改 Cron、参数等） |
| `{admin}/jobinfo/remove?id={id}` | GET | 删除任务 |
| `{admin}/jobinfo/start?id={id}` | GET | 启动任务 |
| `{admin}/jobinfo/stop?id={id}` | GET | 停止任务 |
| `{admin}/jobinfo/pageList` | POST | 分页查询任务列表 |

**核心参数 `XxlJobInfo`**：

| 字段 | 说明 |
|------|------|
| `jobGroup` | 执行器 ID（对应 `xxl_job_group` 表的主键） |
| `jobDesc` | 任务描述 |
| `executorHandler` | JobHandler 名称（即 Agent 名称） |
| `executorParam` | 任务参数（传递给 Agent 的输入） |
| `scheduleType` | 调度类型：`CRON` / `FIXED_RATE` / `NONE` |
| `scheduleConf` | 调度配置：CRON 表达式 或 固定频率（秒） |
| `glueType` | 任务模式：`BEAN`（默认，使用 JobHandler） |
| `executorRouteStrategy` | 路由策略：`FIRST` / `ROUND` / `RANDOM` 等 |
| `executorBlockStrategy` | 阻塞策略：`SERIAL_EXECUTION`（默认） |
| `executorTimeout` | 超时时间（秒） |
| `executorFailRetryCount` | 失败重试次数 |

##### @Tool 工具层实现

在自动登录的 `XxlJobAdminClient` 基础上，`@Tool` 工具类变得非常简洁：

```java
@Component
public class XxlJobScheduleTools {

    private final XxlJobAdminClient client;

    public XxlJobScheduleTools(XxlJobAdminClient client) {
        this.client = client;
    }

    @Tool(name = "xxl_create_scheduled_task",
          description = "创建一个定时任务，支持指定 Cron 表达式，通过 XXL-Job 分布式调度执行")
    public String createScheduledTask(
            @ToolParam("任务名称（JobHandler 名称），需唯一") String taskName,
            @ToolParam("任务描述") String description,
            @ToolParam("Cron 表达式，如 0 0 8 * * ? 表示每天8点") String cronExpression,
            @ToolParam("任务参数（可选，传递给 Agent 的输入内容）") String taskParam
    ) {
        Map<String, String> params = Map.of(
            "jobGroup",              "1",
            "jobDesc",               description,
            "executorHandler",       taskName,
            "executorParam",         taskParam != null ? taskParam : "",
            "scheduleType",          "CRON",
            "scheduleConf",          cronExpression,
            "glueType",              "BEAN",
            "executorRouteStrategy", "ROUND",
            "executorBlockStrategy", "SERIAL_EXECUTION",
            "executorTimeout",       "60",
            "executorFailRetryCount","0"
        );

        JSONObject result = client.addJob(params);
        int jobId = result.getIntValue("content");

        // 创建后自动启动
        client.startJob(jobId);

        return "定时任务创建成功，任务ID: " + jobId
             + "，名称: " + taskName
             + "，Cron: " + cronExpression;
    }

    @Tool(name = "xxl_update_scheduled_task",
          description = "修改定时任务的调度策略（如修改 Cron 表达式）")
    public String updateScheduledTask(
            @ToolParam("XXL-Job 任务ID") int jobId,
            @ToolParam("新的 Cron 表达式") String newCronExpression
    ) {
        Map<String, String> params = Map.of(
            "id", String.valueOf(jobId),
            "scheduleType", "CRON",
            "scheduleConf", newCronExpression
        );
        client.updateJob(params);
        return "任务 " + jobId + " 的 Cron 已更新为: " + newCronExpression;
    }

    @Tool(name = "xxl_stop_scheduled_task", description = "停止一个正在运行的定时任务")
    public String stopTask(@ToolParam("XXL-Job 任务ID") int jobId) {
        client.stopJob(jobId);
        return "任务 " + jobId + " 已停止";
    }

    @Tool(name = "xxl_start_scheduled_task", description = "启动一个已停止的定时任务")
    public String startTask(@ToolParam("XXL-Job 任务ID") int jobId) {
        client.startJob(jobId);
        return "任务 " + jobId + " 已启动";
    }

    @Tool(name = "xxl_delete_scheduled_task", description = "删除一个定时任务")
    public String deleteTask(@ToolParam("XXL-Job 任务ID") int jobId) {
        client.removeJob(jobId);
        return "任务 " + jobId + " 已删除";
    }

    @Tool(name = "xxl_list_scheduled_tasks", description = "查询当前所有定时任务列表")
    public String listTasks() {
        JSONObject result = client.pageList(Map.of("start", "0", "length", "100"));
        return result.toJSONString();
    }
}
```

> 用户：每天早上 8 点帮我生成一份项目日报
> Agent：（调用 `xxl_create_scheduled_task`）→ 自动登录 → POST `/jobinfo/add` → GET `/jobinfo/start`
> Agent：好的，已通过 XXL-Job 创建并启动了定时任务，每天早上 8 点自动执行。

> 用户：把日报改为每天下午 5 点
> Agent：（调用 `xxl_update_scheduled_task`）→ POST `/jobinfo/update`
> Agent：已将日报任务的 Cron 表达式更新为 `0 0 17 * * ?`。

> 用户：暂停日报任务
> Agent：（调用 `xxl_stop_scheduled_task`）→ GET `/jobinfo/stop?id=123`
> Agent：已暂停任务。

#### 方案二：直接操作 XXL-Job 数据库

XXL-Job 的任务配置存储在 `xxl_job_info` 表中，理论上可以直接通过 SQL 插入/更新记录：

```sql
-- 核心表结构
CREATE TABLE xxl_job_info (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    job_group       INT NOT NULL COMMENT '执行器ID',
    job_desc        VARCHAR(255) NOT NULL,
    schedule_type   VARCHAR(50) NOT NULL DEFAULT 'NONE' COMMENT 'CRON/FIXED_RATE/NONE',
    schedule_conf   VARCHAR(128) DEFAULT NULL COMMENT 'Cron表达式或固定频率',
    glue_type       VARCHAR(50) NOT NULL COMMENT 'BEAN/GLUE...',
    executor_handler VARCHAR(255) DEFAULT NULL COMMENT 'JobHandler名称',
    executor_param  VARCHAR(512) DEFAULT NULL COMMENT '任务参数',
    executor_route_strategy VARCHAR(50) DEFAULT NULL,
    executor_block_strategy  VARCHAR(50) DEFAULT NULL,
    executor_timeout INT NOT NULL DEFAULT '0',
    executor_fail_retry_count INT NOT NULL DEFAULT '0',
    trigger_status  TINYINT NOT NULL DEFAULT '0' COMMENT '0-停止 1-运行',
    -- ... 其他字段
);

-- 直接插入任务
INSERT INTO xxl_job_info (job_group, job_desc, schedule_type, schedule_conf,
    executor_handler, executor_param, glue_type, trigger_status)
VALUES (1, '每日日报', 'CRON', '0 0 8 * * ?',
    'daily_report_agent', '', 'BEAN', 1);
```

**不推荐此方案**，原因：
- 需要额外维护数据库连接和 SQL
- 跳过 XXL-Job 的业务逻辑校验，可能产生脏数据
- 调度中心有缓存机制，直接改库可能不会立即生效，需手动触发刷新
- 后续 XXL-Job 版本升级可能导致表结构变更，维护成本高

#### 方案三：扩展 AdminBiz 接口（需二次开发 XXL-Job）

在 XXL-Job 源码中扩展 `AdminBiz` 接口，添加 `addJob`/`updateJob`/`removeJob`/`startJob`/`stopJob` 等方法，通过 `accessToken` 认证调用，无需 Cookie 登录：

```java
// 扩展 AdminBiz 接口
public interface AdminBiz {
    // 已有的接口...
    ReturnT<String> addJob(XxlJobInfo jobInfo);
    ReturnT<String> updateJob(XxlJobInfo jobInfo);
    ReturnT<String> removeJob(int id);
    ReturnT<String> startJob(int id);
    ReturnT<String> stopJob(int id);
}

// AdminBizClient 实现（通过 accessToken 认证）
@Override
public ReturnT<String> addJob(XxlJobInfo jobInfo) {
    return XxlJobRemotingUtil.postBody(
        addressUrl + "api/addJob", accessToken, timeout, jobInfo, String.class);
}

// AdminBizImpl 实现（注入已有的 XxlJobService）
@Override
public ReturnT<String> addJob(XxlJobInfo jobInfo) {
    return xxlJobService.add(jobInfo);
}
```

此方案最规范，但需要修改 XXL-Job 源码并重新部署 Admin 服务。

#### 三种方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Admin HTTP 接口** | 无需修改 XXL-Job 源码；开箱即用；社区广泛使用 | 需要登录获取 Cookie；非官方公开 API | **推荐** |
| **直接操作数据库** | 最直接，不依赖 HTTP 调用 | 缓存一致性风险；跳过校验；维护成本高 | 不推荐 |
| **扩展 AdminBiz** | 最规范；支持 accessToken 认证 | 需二次开发 XXL-Job；需重新部署 Admin | 适合深度定制场景 |

### 修正后的能力对比

结合 XXL-Job Admin API 方案后，两种调度器的对话式能力对比如下：

| 调度器 | 对话式创建 | 对话式启停 | 对话式删除 | 对话式修改Cron | 实现复杂度 |
|--------|-----------|-----------|-----------|--------------|-----------|
| **Quartz** | 直接支持 | 直接支持 | 直接支持 | 直接支持 | 低（内置 API） |
| **XXL-Job** | 支持（Admin HTTP） | 支持（Admin HTTP） | 支持（Admin HTTP） | 支持（Admin HTTP） | 中（需封装 HTTP 客户端） |

> **结论**：XXL-Job 完全可以实现智能体对话式动态创建定时任务，只需在 `@Tool` 工具类中封装 XXL-Job Admin 的 HTTP 接口即可。当前 `XxlJobAgentScheduler` 的限制（不支持 cancel、仅支持 NONE 模式）是 **AgentScope 调度器扩展层的设计选择**，而非 XXL-Job 本身的能力限制。通过 Admin HTTP API 可以绕过这层限制，实现完整的对话式任务生命周期管理。

## 五、定时任务执行效果与结果输出

### 执行流程概览

```
调度器触发
    │
    ▼
BaseScheduleAgentTask.run()
    │
    ├── 检查 cancelled 标志（已取消则抛异常）
    │
    ▼
createAgent() ── 每次创建全新的 ReActAgent 实例
    │             （使用 RuntimeAgentConfig 中的 model/toolkit/hooks/sysPrompt）
    ▼
agent.call(msgs) ── ReAct 循环执行
    │                ├── 推理（Reasoning）：调用 LLM 获取响应
    │                └── 行动（Acting）：执行 @Tool 方法（如有）
    ▼
返回 Mono<Msg> ── Agent 的最终响应消息
    │
    ▼
.block() 阻塞等待结果
    │
    ▼
执行计数 +1（executionCount.incrementAndGet()）
```

**核心效果**：定时任务的本质是 **周期性地自动执行一次 ReActAgent 的完整推理-行动循环**。Agent 会按照 `sysPrompt` 中定义的流程，自主调用绑定的 `@Tool` 工具完成业务逻辑，最终产出一个 `Msg` 响应。

### 结果输出到哪里？

**`BaseScheduleAgentTask.run()` 的返回值 `Mono<Msg>` 就是 Agent 的执行结果**，但这个返回值在不同调度器中的处理方式不同：

#### Quartz：结果在调度线程中消费

```java
// AgentQuartzJob.execute() 中的关键代码
t.run().block();  // 阻塞等待 Agent 执行完成，返回值被丢弃
```

Quartz 实现中，`AgentQuartzJob` 调用 `task.run().block()` 后**返回值没有被使用**。这意味着 Agent 的输出结果完全依赖于以下两种方式传递：

1. **通过 Hook 捕获**：在 `RuntimeAgentConfig` 中配置 Hook，拦截 Agent 的推理/行动事件
2. **通过 @Tool 工具副作用**：Agent 在执行过程中调用 `@Tool` 方法产生副作用（写文件、发通知、写数据库等）

#### XXL-Job：结果自动记录到日志

```java
// XxlJobAgentScheduler 自动添加的 Hook
new Hook() {
    @Override
    public <T extends HookEvent> Mono<T> onEvent(T event) {
        if (event instanceof PostReasoningEvent postReasoningEvent) {
            Msg msg = postReasoningEvent.getReasoningMessage();
            XxlJobHelper.log(msg.getTextContent());  // Agent 输出写入 XXL-Job 日志
        }
        return Mono.just(event);
    }
};

// 执行完成后标记成功
XxlJobHelper.handleSuccess("Successfully executed scheduled task '" + name + "'");
```

XXL-Job 实现中，调度器**自动添加了一个 `PostReasoningEvent` Hook**，将 Agent 每次推理的输出通过 `XxlJobHelper.log()` 写入 XXL-Job 日志系统，可在 Admin 控制台查看。执行完成后调用 `XxlJobHelper.handleSuccess()` 标记任务成功。

### 结果输出的三种模式

| 输出模式 | 说明 | 适用场景 |
|----------|------|----------|
| **@Tool 工具副作用** | Agent 在执行过程中调用 @Tool 方法，在方法内完成实际业务操作（写数据库、发消息、生成文件等） | 最常用，所有实际业务场景 |
| **Hook 事件拦截** | 通过 PreReasoning/PostReasoning/PreActing/PostActing 等 Hook 拦截 Agent 的事件流，记录日志或触发回调 | 监控、审计、日志收集 |
| **run() 返回值** | `Mono<Msg>` 是 Agent 的最终响应，包含文本/图片/工具调用结果等 | 手动触发场景，编程式获取结果 |

### 实际案例：boba-tea-shop 日报定时任务

以项目中的 `boba-tea-shop` 示例为例，完整展示定时任务的效果和输出：

**配置**（`ScheduleAgentConfig.java`）：
```java
// 注册日报 Agent，绑定 ScheduleAgentTools 工具集
RuntimeAgentConfig agentConfig = RuntimeAgentConfig.builder()
    .name("daily_report_agent")
    .sysPrompt(dailyReportAgentInstruction)  // 指导 Agent 的执行流程
    .toolkit(toolkit)                        // 注册了 ScheduleAgentTools
    .modelConfig(modelConfig)
    .build();

agentScheduler.schedule(agentConfig, scheduleConfig);
```

**Agent 执行流程**（由 sysPrompt 指导）：

```
XXL-Job 定时触发
    │
    ▼
创建 ReActAgent（fresh instance）
    │
    ▼
Agent 推理："我需要先生成日报..."
    │
    ▼
调用 @Tool: getDailyReportInfo()  ←──── 工具1：查询业务数据
    │  ├── 查询 OrderMapper（订单数据）
    │  ├── 查询 FeedbackMapper（反馈数据）
    │  └── 查询 ProductMapper（商品数据）
    │  返回：{total_sales, total_revenue, top3_products, ratings, ...}
    │
    ▼
Agent 推理："数据已获取，生成 Markdown 日报..."
    │
    ▼
调用 @Tool: sendReport(reportText)  ←──── 工具2：输出结果
    │  ├── saveReportToFile(text)     → 保存为 reports/business_report_20260421_080000.md
    │  └── DingTalk webhook POST     → 发送到钉钉群机器人
    │  返回："发送成功"
    │
    ▼
Agent 完成，返回最终 Msg
    │
    ▼
XXL-Job 标记任务成功，日志可在控制台查看
```

**输出结果**：

| 输出渠道 | 内容 | 位置 |
|----------|------|------|
| 本地文件 | Markdown 格式的业务日报 | `{user.dir}/reports/business_report_{yyyyMMdd_HHmmss}.md` |
| 钉钉群消息 | Markdown 格式的业务日报 | 钉钉群聊（通过机器人 Webhook） |
| XXL-Job 日志 | Agent 每次推理的文本输出 | XXL-Job Admin 控制台 → 调度日志 |

### 总结

定时器的核心效果是 **让 Agent 周期性地自主完成一个完整的 ReAct 推理-行动循环**。结果的输出不是由调度器框架直接处理的，而是由 **Agent 绑定的 @Tool 工具**来决定输出到哪里（文件、消息推送、数据库、API 调用等）。调度器框架本身只负责：
- **触发执行**（按 Cron/固定频率/固定延迟）
- **状态管理**（暂停/恢复/取消）
- **日志记录**（XXL-Job 自动记录，Quartz 需自行通过 Hook 实现）

## 六、核心设计要点

1. **状态隔离**：每次调度执行都会动态创建全新的 `ReActAgent` 实例，不共享状态，避免并发问题。
2. **两层配置**：`AgentConfig`（可序列化，用于持久化/分布式场景）与 `RuntimeAgentConfig`（包含 `Toolkit`/`Hook` 等运行时对象，用于同 JVM 场景）。
3. **执行计数**：`BaseScheduleAgentTask` 通过 `AtomicLong` 跟踪每个任务的执行次数。
4. **取消机制**：通过 `AtomicBoolean cancelled` 标志位实现协作式取消，已取消的任务调用 `run()` 会抛出异常。
5. **结果输出**：调度器不直接处理 Agent 的输出结果，而是通过 `@Tool` 工具的副作用和 Hook 事件拦截来实现结果的持久化和推送。
