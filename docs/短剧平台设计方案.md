# 短剧 AI 生成平台 — 基于AgentScope Java的设计方案

## 一、需求分析

### 1.1 竞品流水线分析（纳米漫剧 namistory.com）

纳米漫剧采用 **7步流水线** 模式，每步可独立修改、迭代：

```
用户输入（题材/风格/简介）
  │
  ├─ Step 1: 全局设定 ─── 画风、时代背景、角色数量、时长
  ├─ Step 2: 故事剧本 ─── 完整剧本（场景、对话、旁白）
  ├─ Step 3: 场景角色道具 ─── 每个角色/场景/道具的视觉设定图
  ├─ Step 4: 分镜脚本 ─── 按镜头拆分（景别、运镜、台词、动作描述）
  ├─ Step 5: 分镜视频 ─── 每个镜头生成视频片段
  ├─ Step 6: 配音对口型 ─── TTS生成配音 + 口型同步
  └─ Step 7: 视频预览 ─── 合成最终成片 + 预览
```

**核心特征**：
- **流程化但可分步修改**：用户可在任意步骤返回修改，后续步骤联动更新
- **每步输出结构化数据**：剧本是JSON/结构文本，角色是图片，分镜是列表
- **多模态混合**：文本生成 → 图片生成 → 视频生成 → 音频生成
- **人机交互**：每步结果都需要人工审核/修改后才进入下一步

### 1.2 核心需求

| 维度 | 需求 |
|------|------|
| 流程编排 | 多步骤串行流水线，支持跳步、回退、分支 |
| 多智能体 | 每个步骤由专门的Agent负责（编剧、美术、导演、剪辑） |
| 多模态生成 | 文生图、文生视频、图生视频、TTS、视频理解 |
| 上下文管理 | 项目状态持久化，各步骤间数据传递 |
| 人机交互 | 每步结果可审核、修改后继续 |
| 流式输出 | 前端实时展示生成进度 |

## 四、核心流程设计

### 4.1 流水线模式选择

采用 **Supervisor + SubAgentTool** 模式（推荐），而非 SequentialPipeline：

| 维度 | Supervisor + SubAgentTool | SequentialPipeline |
|------|--------------------------|--------------------|
| 灵活调度 | 总导演Agent可灵活决定执行顺序 | 固定线性执行 |
| 分步修改 | 通过session_id恢复任意步骤 | 需要重新执行整个管道 |
| 人机交互 | 每个子Agent可独立加HITL Hook | 只能在管道级别加 |
| 错误恢复 | 单步失败可重试 | 管道中断需从头开始 |
| 上下文传递 | 通过Memory + Session | 通过Msg传递 |

### 4.3 跳步机制 — 手动导入已有素材

用户可以跳过任意步骤，直接用已有内容填充该步骤，后续步骤会自动读取。

**典型场景**：
- 已有完整剧本 → 跳过Step 2，直接导入剧本
- 已有角色设定图 → 跳过Step 3，直接上传图片
- 已有分镜脚本 → 跳过Step 4，直接导入

```java
/**
 * 手动填充某个步骤的内容（跳过AI生成）
 * @param projectId 项目ID
 * @param step 步骤号 (1-7)
 * @param content 用户提供的结构化内容（JSON/文件URL）
 */
public void importStepContent(String projectId, int step, String content) {
    String stepSessionKey = projectId + "_step_" + step;

    // 直接将用户内容保存到Session，标记为 APPROVED
    SessionKey key = new SessionKey(stepSessionKey);
    session.save(key, "step_result", new StepResult(content, StepStatus.APPROVED));
    session.save(key, "step_meta", new StepMeta(
        step, StepStatus.APPROVED, Instant.now(), "USER_IMPORTED"
    ));
}

/**
 * 支持多种导入格式
 */
public void importStepContent(String projectId, int step, MultipartFile file) {
    // 将上传的文件存储到OSS，拿到URL后保存
    String fileUrl = mediaStorageService.upload(file);
    importStepContent(projectId, step, fileUrl);
}

/**
 * 批量导入 — 一键填充多个步骤
 */
public void batchImport(String projectId, Map<Integer, String> stepContents) {
    stepContents.forEach((step, content) -> importStepContent(projectId, step, content));
}
```

### 4.4 步骤进度追踪 — 实时查看每步完成情况

每个步骤有独立的状态，前端可以随时查询整个项目的进度看板。

**步骤状态模型**：

```java
public enum StepStatus {
    EMPTY,          // 未开始，无数据
    IMPORTED,       // 用户手动导入
    GENERATING,     // AI正在生成中
    WAITING_REVIEW, // 生成完毕，等待用户审核
    APPROVED,       // 用户已确认通过
    FAILED          // 生成失败
}
```

**进度查询 API**：

```java
@RestController
@RequestMapping("/api/project/{projectId}")
public class ProgressController {

    /**
     * 获取项目所有步骤的进度看板
     * GET /api/project/{projectId}/progress
     */
    @GetMapping("/progress")
    public ProjectProgress getProgress(@PathVariable String projectId) {
        return dramaService.getProjectProgress(projectId);
    }

    /**
     * SSE 实时推送进度变化
     * GET /api/project/{projectId}/progress/stream
     */
    @GetMapping(value = "/progress/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<StepProgress>> streamProgress(
            @PathVariable String projectId) {
        return dramaService.streamProgressUpdates(projectId);
    }
}
```

**进度数据结构**：

```java
public class ProjectProgress {
    private String projectId;
    private String projectTitle;
    private List<StepProgress> steps;

    /** 当前可以执行的下一步（依赖前序步骤是否完成） */
    private int nextExecutableStep;

    /** 整体完成百分比 */
    private int completionPercentage;
}

public class StepProgress {
    private int stepNumber;
    private String stepName;        // "全局设定" / "故事剧本" / ...
```

**步骤可执行条件判断**：

```java
/**
 * 判断某步骤是否可以执行
 * 规则：前序所有步骤状态为 APPROVED 或 IMPORTED
 */
public boolean isStepExecutable(String projectId, int step) {
    if (step == 1) return true; // Step 1随时可执行

    // 检查前序步骤是否都已完成
    for (int i = 1; i < step; i++) {
        StepStatus status = getStepStatus(projectId, i);
        if (status != StepStatus.APPROVED && status != StepStatus.IMPORTED) {
            return false;
        }
    }
    return true;
}

/**
 * 自动计算下一步可执行的步骤
 */
public int getNextExecutableStep(String projectId) {
    for (int i = 1; i <= 7; i++) {
        StepStatus status = getStepStatus(projectId, i);
        if (status == StepStatus.EMPTY || status == StepStatus.FAILED) {
            return isStepExecutable(projectId, i) ? i : -1;
        }
    }
    return -1; // 全部完成
}
```

### 4.5 分步执行与回退机制

**核心思路**：将整个流水线拆为7个独立步骤，每步使用独立的session_id存储状态。

```java
public class DramaProjectService {

    private final Session session;
    private final Map<String, ReActAgent> projectAgents = new ConcurrentHashMap<>();

    /**
     * 执行指定步骤
     * @param projectId 项目ID
     * @param step 步骤号 (1-7)
     * @param userInput 用户输入/修改内容（可选）
     */
    public Flux<Event> executeStep(String projectId, int step, String userInput) {
        // 前置检查：前序步骤必须已完成
        if (!isStepExecutable(projectId, step)) {
            throw new IllegalStateException(
                "步骤" + step + "的前序步骤尚未完成，请先完成前置步骤或手动导入内容");
        }

        String stepSessionKey = projectId + "_step_" + step;

        // 更新步骤状态为 GENERATING
        updateStepStatus(projectId, step, StepStatus.GENERATING);

        // 根据步骤号选择对应的子智能体
        ReActAgent agent = getAgentForStep(step);

        // 构造输入消息（包含前面步骤的产出）
        String context = buildContextFromPreviousSteps(projectId, step);
        Msg input = Msg.builder()
            .role(MsgRole.USER)
            .textContent(context + "\n\n" + (userInput != null ? userInput : "请执行此步骤"))
            .build();

        // 流式执行
        return agent.call(input)
            .flatMapMany(response -> agent.stream(response, StreamOptions.DEFAULT))
            .doOnComplete(() -> {
                // 保存当前步骤状态，标记为等待审核
                agent.saveTo(session, new SessionKey(stepSessionKey));
                updateStepStatus(projectId, step, StepStatus.WAITING_REVIEW);
            })
            .doOnError(e -> {
                updateStepStatus(projectId, step, StepStatus.FAILED);
            });
    }

    /**
     * 修改已完成的步骤，并级联更新后续步骤
     */
    public Flux<Event> modifyStep(String projectId, int step, String modification) {
        // 1. 清除当前步骤及后续步骤的状态
        for (int i = step; i <= 7; i++) {
            session.delete(new SessionKey(projectId + "_step_" + i));
            updateStepStatus(projectId, i, StepStatus.EMPTY);
        }

        // 2. 用修改内容重新执行当前步骤
        return executeStep(projectId, step, modification);
    }

    /**
     * 从前序步骤构建上下文（包括手动导入的内容）
     */
    private String buildContextFromPreviousSteps(String projectId, int currentStep) {
        StringBuilder context = new StringBuilder();

        for (int i = 1; i < currentStep; i++) {
            StepStatus status = getStepStatus(projectId, i);
            // 无论 AI生成 还是 手动导入 的内容，都读取
            if (status == StepStatus.APPROVED || status == StepStatus.IMPORTED) {
                context.append("## ").append(getStepName(i)).append("\n")
                    .append(loadStepResult(projectId, i)).append("\n\n");
            }
        }

        return context.toString();
    }
}
```

```java
 * 步骤审核Hook — 每个步骤执行完毕后暂停，等待用户确认

public class StepReviewHook implements Hook {

    @Override
    public <T extends HookEvent> Mono<T> onEvent(T event) {
        if (event instanceof PostReasoningEvent pre) {
            Msg msg = pre.getReasoningMessage();
            // 检查是否包含步骤完成的标记
            if (msg != null && containsStepCompletion(msg)) {
                // 暂停Agent，将结果返回给前端
                pre.stopAgent();
            }
        }
        return Mono.just(event);
    }
}
```

## 五、多智能体上下文管理机制

### 5.1 一句话总结

**每个 Agent 实例有自己私有的 Memory，父子 Agent 之间只传纯文本，跨实例的状态靠 Session 桥接。**

### 5.2 上下文隔离的三层架构

```
┌─────────────────────────────────────────────────────┐
│ 第一层：实例隔离                                       │
│   每次调用 SubAgentTool 时，agentProvider.provide()   │
│   创建全新 Agent 实例（独立 Memory / Toolkit / sysPrompt）│
├─────────────────────────────────────────────────────┤
│ 第二层：纯文本通信                                     │
│   子Agent 返回 ToolResultBlock.text("结果摘要")       │
│   父Agent 的 Memory 只追加一条文本，不会膨胀             │
├─────────────────────────────────────────────────────┤
│ 第三层：Session 桥接                                   │
│   同一 session_id 的多次调用，通过 Session 恢复完整状态   │
│   JsonSession 持久化到文件，关闭浏览器后状态不丢失        │
└─────────────────────────────────────────────────────┘
```

### 5.3 第一层：实例隔离（不是上下文切换）

AgentScope **不搞虚拟上下文切换**，而是每次调用都创建新实例：

```java
// 不是这样（虚拟上下文 — AgentScope不采用）:
agent.switchContext("screenwriter_session");
agent.call(msg);

// 而是这样（实例隔离 — AgentScope实际做法）:
Agent instance = agentProvider.provide();  // 全新实例
instance.loadFrom(session, sessionId);     // 从Session恢复之前状态
instance.call(msg);                        // 执行
instance.saveTo(session, sessionId);       // 保存状态
```

每个实例的 Memory 是 `CopyOnWriteArrayList<Msg>`，绑定到实例，运行时不可替换：

```java
// ReActAgent 构造函数
this.memory = builder.memory;

// 运行时替换直接抛异常
public void setMemory(Memory memory) {
    throw new UnsupportedOperationException(
        "Memory cannot be replaced after agent construction.");
}
```

### 5.4 第二层：父子纯文本通信

子Agent的完整创作过程不会泄漏到父Agent的Memory中：

```
Director（总导演）Memory:
  [SYSTEM]    你是总导演...
  [USER]      生成都市爱情短剧
  [ASSISTANT] 调用编剧...                  ← ToolUseBlock: call_screenwriter
  [TOOL]      session_id: abc\n\n剧本摘要... ← 只有纯文本！子Agent的完整创作过程不在这里
  [ASSISTANT] 调用美术指导...
  [TOOL]      角色A图: url1, 角色B图: url2  ← 只有URL文本
  [ASSISTANT] 汇总结果

Screenwriter（编剧）Memory — 完全隔离在另一个实例:
  [SYSTEM]    你是专业编剧...
  [USER]      写一个都市爱情剧本
  [ASSISTANT] 故事大纲：...（思考过程）
  [USER]      修改第二幕
  [ASSISTANT] 已修改...
  [ASSISTANT] 最终剧本：...

ArtDirector（美术）Memory — 又是另一个独立实例:
  [SYSTEM]    你是美术指导...
  [USER]      根据以下剧本设计角色...
  [ASSISTANT] 调用 dashscope_text_to_image
  [TOOL]      ImageBlock(url=...)           ← 图片URL在美术自己的Memory里
  [ASSISTANT] 已生成3个角色设定图
```

**SubAgentTool 返回给父Agent的内容**：

```java
// SubAgentTool.callAsync() 内部
return ToolResultBlock.text(
    "session_id: " + sessionId + "\n\n" + resultContent
);
// resultContent 是子Agent最终回复的纯文本，不是完整的Memory
```

### 5.5 第三层：Session 跨实例状态桥梁

通过 `session_id` + Session 实现同一子Agent的多轮对话：

```
第1次调用（新会话）:
  call_screenwriter(message="写剧本", session_id=null)

  SubAgentTool内部:
  1. session_id == null → isNewSession = true → 生成 UUID "scr-001"
  2. agentProvider.provide() → 新ScreenwriterAgent实例（空Memory）
  3. isNewSession → 跳过 loadFrom
  4. agent.call(userMsg) → 编剧开始创作，Memory累积
  5. saveTo(session, "scr-001") → 保存 agent_meta + memory_messages + toolkit状态
  6. 返回 ToolResultBlock.text("session_id: scr-001\n\n剧本内容...")

第2次调用（继续会话）:
  call_screenwriter(message="改结局为大团圆", session_id="scr-001")

  SubAgentTool内部:
  1. session_id == "scr-001" → isNewSession = false
  2. agentProvider.provide() → 又一个新ScreenwriterAgent实例（空Memory）
  3. agent.loadFrom(session, "scr-001") → 从Session恢复完整对话历史
  4. agent.call(userMsg) → 编剧看到之前所有上下文，继续修改
  5. saveTo(session, "scr-001") → 增量保存更新后的Memory
  6. 返回 ToolResultBlock.text("session_id: scr-001\n\n修改后的剧本...")
```

Session 保存的内容：

```
实例 A (第1次调用)            Session 存储               实例 B (第2次调用)
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│ memory: [...] │ ──→     │ agent_meta   │  ──→    │ memory: [...] │  ← 恢复完整对话
│ toolkit: {...}│ ──→     │ memory_msgs  │  ──→    │ toolkit: {...}│
│ plan: {...}   │ ──→     │ toolkit_grps │  ──→    │ plan: {...}   │
└──────────────┘         └──────────────┘         └──────────────┘
```

### 5.6 各种多智能体模式的上下文对比

| 模式 | 隔离方式 | Memory行为 | 状态持久化 | 通信方式 |
|------|---------|-----------|-----------|---------|
| **SubAgentTool（工厂模式）** | `SubAgentProvider` 每次新建实例 | 各自独立Memory | Session（InMemory/Json） | ToolResultBlock纯文本 |
| **Supervisor（共享实例）** | 同一实例 | Memory跨调用累积 | 无 | ToolResultBlock纯文本 |
| **SequentialPipeline** | 独立实例 | 上一步输出→下一步输入 | 无 | Msg传递 |
| **FanoutPipeline** | 独立实例 | 同一输入广播到所有Agent | 无 | 各自独立返回List |
| **MsgHub** | observe广播 | 各自Memory独立累积 | 无 | Msg广播 |
| **StateGraph/Handoffs** | 图状态共享 | 图状态中的messages | 无 | 图状态变量驱动路由 |

### 5.7 短剧场景的上下文流转全链路

```
用户: "生成都市爱情短剧"
  │
  ▼
DramaDirector（总导演）
  Memory: [SYSTEM:总导演提示, USER:用户需求]
  │
  ├── call_screenwriter("写剧本", session_id=null)
  │     新实例 → 编剧独立Memory累积创作过程
  │     saveTo(session, "scr-001")
  │     返回纯文本 → 导演Memory追加: [TOOL_RESULT: "剧本摘要..."]
  │
  ├── call_screenwriter("改结局为大团圆", session_id="scr-001")
  │     新实例 → loadFrom("scr-001") → 恢复编剧之前的完整对话
  │     编剧看到所有历史，继续修改
  │     saveTo(session, "scr-001") → 增量保存
  │     返回纯文本 → 导演Memory追加: [TOOL_RESULT: "修改后的剧本..."]
  │
  ├── call_art_director("设计角色", session_id=null)
  │     新实例 → 美术独立Memory，调文生图工具
  │     角色图URL在美术的Memory里
  │     saveTo(session, "art-001")
  │     返回纯文本 → 导演Memory追加: [TOOL_RESULT: "角色A: url1..."]
  │
  ├── call_video_producer("生成视频", session_id=null)
  │     新实例 → 视频Memory，调图生视频工具
  │     saveTo(session, "vid-001")
  │     返回纯文本 → 导演Memory追加: [TOOL_RESULT: "镜头1: video_url..."]
  │
  └── 导演Memory最终状态:
      [用户需求] → [编剧结果] → [修改编剧结果] → [美术结果] → [视频结果] → [汇总]
       ↑ 从头到尾没有子Agent的冗长中间过程，不会上下文溢出
```

### 5.8 在短剧各步骤中的具体应用

| 场景 | 上下文管理方式 | 涉及的AgentScope机制 |
|------|--------------|--------------------|
| 总导演调度子Agent | 每个子Agent独立Memory，结果以纯文本回传 | SubAgentTool + ToolResultBlock.text() |
| 编剧多轮修改剧本 | `session_id` 保持同一创作会话 | Session + loadFrom/saveFrom |
| Step 3读取Step 2的剧本 | 从Session读取前步结果注入Agent输入 | `buildContextFromPreviousSteps()` |
| 用户修改Step 2后级联更新 | 清除Step 2~7的Session，后续重新生成 | Session.delete() |
| 用户关闭浏览器后回来 | JsonSession文件持久化，重新加载 | JsonSession（文件系统） |
| 角色一致性跨镜头保持 | 角色设定图URL保存在Session中，每步读取 | Session存储 + Prompt模板注入 |
| 多镜头视频并行生成 | FanoutPipeline，每个镜头独立实例 | FanoutPipeline.concurrent() |

### 5.9 设计原则总结

| 原则 | 说明 | 短剧场景的体现 |
|------|------|--------------|
| 上下文隔离靠实例隔离 | 不做虚拟上下文切换，而是创建新实例 | 每个步骤由独立Agent实例处理 |
| Memory是私有状态 | 构建时绑定，运行时不可替换 | 编剧的创作过程不会出现在导演的Memory里 |
| Session是状态桥梁 | 跨实例序列化/反序列化 | 用户关闭后重开，项目状态完整恢复 |
| 父子只传纯文本 | ToolResultBlock.text() | 导演只看到"剧本内容..."而非编剧的50轮对话 |
| 线程安全靠实例隔离 | AtomicBoolean running防并发 | 工厂模式每次新建实例，支持并行 |

## 九、关键技术实现

### 8.1 视频生成策略

每个分镜的视频生成采用 **关键帧 + 图生视频** 的两步策略，而非直接文生视频，以保证画面质量和角色一致性：

```java
/**
 * 分镜视频生成工具 — 封装两步生成策略
 */
@Component
public class ShotVideoGenerator {

    private final DashScopeMultiModalTool multiModalTool;
    private final String apiKey;

    /**
     * 为单个分镜生成视频
     * Step 1: 文生图 → 关键帧
     * Step 2: 图生视频 → 最终镜头
     */
    public ShotVideo generateShotVideo(StoryboardShot shot, CharacterDesign character) {
        // Step 1: 生成关键帧图片
        String imagePrompt = buildImagePrompt(shot, character);
        ToolResultBlock imageResult = multiModalTool.dashscopeTextToImage(
            imagePrompt,
            "wanx-v1",    // 文生图模型
            1,
            "1024*1024",
            false
        ).block();

        String imageUrl = extractImageUrl(imageResult);

        // Step 2: 基于关键帧生成视频
        String videoPrompt = buildVideoPrompt(shot);
        ToolResultBlock videoResult = multiModalTool.dashscopeImageToVideo(
            videoPrompt,
            "wan2.6-i2v-flash",   // 图生视频模型
            imageUrl,
            null,    // audio_url
            null,    // negative_prompt
            null,    // template
            "1080P", // resolution
            (int) shot.getDuration(),  // duration
            "single",  // shot_type
            true,    // audio
            true,    // prompt_extend
            false,   // watermark
            null     // seed
        ).block();

        String videoUrl = extractVideoUrl(videoResult);

        return new ShotVideo(shot.getShotId(), videoUrl, imageUrl, shot.getDuration());
    }

    private String buildImagePrompt(StoryboardShot shot, CharacterDesign character) {
        return String.format(
            "%s风格，%s，%s镜头，%s，高质量，电影质感，4K",
            character != null ? character.getDescription() : "",
            shot.getVisualPrompt(),
            shot.getShotType(),
            shot.getCameraMovement()
        );
    }

    private String buildVideoPrompt(StoryboardShot shot) {
        return String.format(
            "%s，%s运镜",
            shot.getActionDescription() != null ? shot.getActionDescription() : "轻微运动",
            shot.getCameraMovement()
        );
    }
}
```

### 8.2 角色一致性保持

短剧生成的核心挑战是**角色跨镜头的一致性**。方案：

1. **角色设定图固定**：每个角色只生成一次设定图，后续所有镜头引用该图
2. **IP-Adapter / ControlNet**（如有API支持）：使用角色参考图约束生成
3. **Prompt模板化**：角色的外貌描述作为固定前缀加入每个生成prompt

## 十一、技术风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| 视频生成耗时 | 单镜头30-120秒 | 异步队列+进度展示，FanoutPipeline并行生成 |
| 角色跨镜头不一致 | 画风/外貌不统一 | 固定角色设定图 + Prompt模板 + 参考 ControlNet |
| 上下文溢出 | 剧本太长导致LLM token超限 | 分场景独立处理，长剧本分段生成 |
| API成本 | 大量图片/视频生成费用高 | 缓存已生成素材，按需生成，选择性价比模型 |
| DashScope API限流 | 并发生成被限流 | 请求队列 + 指数退避重试 |
| 对口型精度 | 音视频同步不好 | 预留第三方API接口（如Wav2Lip） |

## 十二、总结

本方案基于 **AgentScope Java** 框架，采用 **Supervisor + SubAgentTool** 架构模式：

1. **总导演Agent** 负责整体调度和流程管理
2. **5个专业子Agent**（编剧、美术、分镜、视频、音频）各司其职
3. **Session持久化** 管理项目状态，支持分步修改和回退
4. **HITL Hook机制** 实现每步人工审核
5. **DashScope多模态工具** 提供文生图、图生视频、TTS等核心能力
6. **SSE流式输出** 让前端实时展示生成进度

这种设计的核心优势是 **解耦和灵活性**：每个Agent可以独立开发、测试、优化；用户可以在任意步骤介入修改；后续可以方便地替换底层模型或添加新的Agent能力。
